# 데이터 무결성

## 무결성 예시

- 회사가 삭제되었는데 채용 공고는 남아있는 경우
- 이메일이 중복으로 등록되는 경우
- 존재하지 않는 유저 role이 할당되는 경우

## 데이터베이스에서 무결성을 강제해야 하는 이유

- 애플리케이션 레벨에서 모든 무결성 검증을 구현하는 것은 어렵고, 실수로 누락될 수 있음
- API 서버, 배치 스크립트, 어드민 도구 앱, 마이그레이션 코드, 직접 DB 조작 등 다양한 진입점에서 데이터가 변경될 수 있기 때문
- DB는 트랜잭션과 제약조건을 통해 무결성을 강제할 수 있음
- 데이터가 깨지면 복구가 어렵고, 사용자 신뢰도 하락, 법적 문제 등 심각한 결과를 초래할 수 있음

## 무결성의 종류

### 개체 무결성(entity integrity): 각 레코드가 고유하게 식별될 수 있도록 하는 것

- Primary Key는 null 불가, 중복 불가, 유니크 인덱스 자동 생성

### 참조 무결성(referential integrity): 테이블 간의 관계가 일관되도록 하는 것

- `Foreign Key`는 참조 대상이 존재해야 하며, 참조 대상이 삭제되면 어떻게 할지(ON DELETE)도 정의할 수 있음
- 예1: 채용 공고가 회사에 참조하는 경우, 회사가 삭제되면 해당 채용 공고도 삭제(ON DELETE CASCADE)하거나, 참조를 null로 설정(ON DELETE SET NULL)할 수 있음
- 예2: 주문이 유저에 참조하는 경우, 유저가 삭제되면 해당 주문도 삭제(ON DELETE CASCADE)하거나, 참조를 null로 설정(ON DELETE SET NULL)할 수 있음
- 예3: 댓글이 게시글에 참조하는 경우, 게시글이 삭제되면 해당 댓글도 삭제(ON DELETE CASCADE)하거나, 참조를 null로 설정(ON DELETE SET NULL)할 수 있음

### 도메인 무결성(domain integrity): 컬럼에 저장될 수 있는 값의 범위를 제한하는 것

- 데이터 타입, 길이, 범위, 패턴 등을 제약할 수 있음
- 나이는 0 이상, 이메일은 특정 패턴, 상태는 'active', 'inactive' 중 하나 등

### 사용자 정의 무결성(user-defined integrity): 애플리케이션 로직에 따른 무결성 (예: 특정 비즈니스 규칙을 만족해야 하는 경우)

- 예: 주문이 100달러 이상이면 VIP 고객으로 분류, 특정 날짜 이후에는 할인 적용 불가 등

## Pydantic 모델 검증과 DB 무결성의 관계

- Pydantic 모델은 애플리케이션 레벨에서 입력 데이터의 유효성을 검증하는 역할을 함
- DB 무결성은 데이터가 실제로 저장될 때 일관성을 보장하는 역할을 함
- Pydantic 모델에서 검증이 통과하더라도, DB 무결성 제약조건에 의해 데이터 저장이 실패할 수 있음 (예: 유니크 제약조건 위반, 외래키 제약조건 위반 등)
- 따라서, 애플리케이션 레벨에서 가능한 검증을 수행하되, DB 무결성 제약조건도 반드시 설정하여 데이터의 일관성을 보장하는 것이 중요함

### Pydantic 검증을 DB 제약조건으로 보완하는 방법

- `__table_args__`: SQLAlchemy 모델에서 테이블 수준의 제약조건을 정의하는 데 사용되는 클래스 변수
- 튜플 형태로 여러 제약조건을 나열할 수 있음
- 예시:

```python
class User(Base):
    __tablename__ = 'users'
    id = Column(Integer, primary_key=True)
    email = Column(String, nullable=False)
    age = Column(Integer, nullable=False)
    __table_args__ = (
        UniqueConstraint('email', name='uq_user_email'),
        CheckConstraint('age >= 0', name='ck_user_age_positive'),
    )
```

- 위 예시에서는 `email` 컬럼에 유니크 제약조건과 `age` 컬럼에 체크 제약조건이 설정되어 있음
- `__table_args__`를 사용하여 테이블 수준의 제약조건을 명확하게 정의함으로써, 데이터베이스에서 무결성을 강제할 수 있음

### ON DELETE

- 참조 무결성 제약조건에서 `ON DELETE` 옵션은 참조 대상이 삭제될 때 어떻게 처리할지를 정의하는 옵션
- `ON DELETE CASCADE`: 참조 대상이 삭제되면 해당 레코드도 자동으로 삭제
- `ON DELETE SET NULL`: 참조 대상이 삭제되면 해당 레코드의 참조 컬럼을 null로 설정
- `ON DELETE RESTRICT`: 참조 대상이 삭제되지 않도록 제한 (기본값)
- `ON DELETE NO ACTION`: 참조 대상이 삭제되면 오류 발생 (RESTRICT와 유사하지만, 트랜잭션이 커밋될 때까지 제약조건 검사가 지연됨)
- 예시:

```python
class JobPost(Base):
    __tablename__ = 'job_posts'
    id = Column(Integer, primary_key=True)
    company_id = Column(Integer, ForeignKey('companies.id', ondelete='CASCADE'))
```

- 위 예시에서는 `company_id` 컬럼이 `companies` 테이블의 `id` 컬럼을 참조하며, 참조 대상이 삭제될 때 해당 레코드도 자동으로 삭제되도록 설정되어 있음

_주의_

- `ON DELETE` 옵션은 데이터베이스 수준에서 참조 무결성을 강제하는 것이므로, 애플리케이션 레벨에서 별도의 삭제 로직을 구현할 필요가 없음
- 잘못 설정된 `ON DELETE` 옵션은 데이터 무결성 문제를 야기할 수 있으므로, 이 자식 레코드가 부모 없이는 의미가 없거나, 부모가 삭제되면 자식도 함께 삭제되어야 하는 경우에 적절히 설정하는 것이 중요함
- 예시: 회사가 삭제되면 해당 회사에 속한 채용 공고도 함께 삭제되어야 하는 경우, `ON DELETE CASCADE` 옵션을 설정하는 것이 적절함

### ON DELETE SET NULL

- 참조 대상이 삭제될 때 해당 레코드의 참조 컬럼을 null로 설정하는 옵션
- 예시:

```python
class JobPost(Base):
    __tablename__ = 'job_posts'
    id = Column(Integer, primary_key=True)
    company_id = Column(
        Integer,
        ForeignKey('companies.id', ondelete='SET NULL'),
        nullable=True,
    )
```

- 위 예시에서는 `company_id` 컬럼이 `companies` 테이블의 `id` 컬럼을 참조하며, 참조 대상이 삭제될 때 해당 레코드의 `company_id` 컬럼이 null로 설정되도록 설정되어 있음
- 이 옵션은 자식 레코드가 부모 레코드 없이도 의미가 있는 경우에 적절하게 사용될 수 있음
- 예시: 채용 공고가 회사에 참조하는 경우, 회사가 삭제되더라도 채용 공고 자체는 여전히 유효한 정보로 남아있어야 하는 경우, `ON DELETE SET NULL` 옵션을 설정하는 것이 적절함

### ON DELETE RESTRICT

- 참조 대상이 삭제되지 않도록 제한하는 옵션 (기본값)
- 예시:

```python
class JobPost(Base):
    __tablename__ = 'job_posts'
    id = Column(Integer, primary_key=True)
    company_id = Column(Integer, ForeignKey('companies.id', ondelete='RESTRICT'))
```

- 위 예시에서는 `company_id` 컬럼이 `companies` 테이블의 `id` 컬럼을 참조하며, 참조 대상이 삭제되지 않도록 제한하는 옵션이 설정되어 있음
- 이 옵션은 자식 레코드가 남아 있는 동안 부모 레코드 삭제를 막고 싶을 때 적절함
- 예시: 회사에 아직 채용 공고가 남아 있다면, 먼저 채용 공고를 정리하기 전까지 회사 삭제를 막고 싶은 경우 `ON DELETE RESTRICT`가 적절함

### 결론

- 애매하면 `ON DELETE RESTRICT`로 시작하는 것이 안전함
- 자식 레코드가 부모 레코드 없이는 의미가 없는 경우 `ON DELETE CASCADE`를 고려할 수 있음
- 자식 레코드가 부모 레코드 없이도 의미가 있는 경우 `ON DELETE SET NULL`을 고려할 수 있음
- 데이터 무결성은 애플리케이션 레벨과 데이터베이스 레벨에서 모두 강제하는 것이 중요하며, `ON DELETE` 옵션을 적절히 설정하여 참조 무결성을 유지하는 것이 중요함

## 제약 조건(Constraint)

- 데이터베이스에서 특정 컬럼이나 컬럼 조합에 대해 유효한 값의 범위를 정의하는 규칙
- 데이터 무결성을 보장하기 위해 사용됨
- 주요 제약 조건 종류:
- `PRIMARY KEY`: 각 레코드를 고유하게 식별하는 컬럼에 설정, null 불가, 중복 불가
- `FOREIGN KEY`: 다른 테이블의 컬럼을 참조하는 컬럼에 설정, 참조 대상이 존재해야 함
- `UNIQUE`: 컬럼의 값이 테이블 내에서 유일해야 함
- `NOT NULL`: 컬럼의 값이 null이 될 수 없음
- `CHECK`: 컬럼의 값이 특정 조건을 만족해야 함
  - `CHECK (status IN ('active', 'inactive'))` → `status` 컬럼의 값이 'active' 또는 'inactive' 중 하나여야 함
  - 예시

```python
class User(Base):
    __tablename__ = 'users'
    id = Column(Integer, primary_key=True)
    email = Column(String, nullable=False)
    age = Column(Integer, nullable=False)
    __table_args__ = (
        UniqueConstraint('email', name='uq_user_email'),
        CheckConstraint('age >= 0', name='ck_user_age_positive'),
    )
```

- 위 예시에서는 `email` 컬럼에 유니크 제약조건과 `age` 컬럼에 체크 제약조건이 설정되어 있음
- 제약 조건은 데이터베이스 수준에서 무결성을 강제하는 역할을 하며, 애플리케이션 레벨에서의 검증과 함께 사용하여 데이터의 일관성을 보장하는 것이 중요함

### 복합 유니크

- 여러 컬럼의 조합이 유일해야 하는 경우에 사용되는 제약 조건
- 예시:

```python
class User(Base):
    __tablename__ = 'users'
    id = Column(Integer, primary_key=True)
    provider = Column(String, nullable=False)
    provider_user_id = Column(String, nullable=False)
    __table_args__ = (
        UniqueConstraint(
            'provider',
            'provider_user_id',
            name='uq_user_provider_identity',
        ),
    )
```

- 위 예시에서는 `provider`, `provider_user_id` 조합이 유일해야 하는 복합 유니크 제약 조건이 설정되어 있음
- 이 제약 조건은 동일한 소셜 로그인 제공자의 동일한 계정이 중복 저장되지 않도록 보장함
- 복합 유니크 제약 조건은 단일 컬럼의 유니크 제약 조건으로는 표현할 수 없는 경우에 유용하게 사용될 수 있음

주의:

- 복합 유니크는 실제 비즈니스 규칙에 맞는 컬럼 조합으로 설계해야 함
- 무조건 컬럼을 많이 묶는다고 좋은 것이 아니라, "무엇이 함께 유일해야 하는가"를 먼저 정의해야 함

### 복합 PK

- 다대다 관계에서 조인 테이블의 경우, 단일 컬럼이 아닌 여러 컬럼의 조합이 기본 키 역할을 하는 경우에 사용되는 제약 조건
- 예시:

```python
class OrderItem(Base):
    __tablename__ = 'order_items'
    order_id = Column(Integer, ForeignKey('orders.id'), primary_key=True)
    product_id = Column(Integer, ForeignKey('products.id'), primary_key=True)
    quantity = Column(Integer, nullable=False)
```

- 위 예시에서는 `order_id`와 `product_id` 컬럼의 조합이 기본 키 역할을 하는 복합 PK 제약 조건이 설정되어 있음
- 이 제약 조건은 동일한 주문(`order_id`)에 동일한 제품(`product_id`)이 여러 번 포함될 수 없도록 보장함
- 복합 PK 제약 조건은 조인 테이블에서 자주 사용되며, 각 레코드가 고유하게 식별될 수 있도록 보장하는 역할을 함

## 정규화

- 데이터베이스 설계에서 데이터 중복을 최소화하고, 데이터 무결성을 높이기 위해 테이블을 분리하는 과정
- 정규화 단계:
- 제1정규형(1NF): 각 컬럼이 원자값을 가지도록 테이블을 설계
- 제2정규형(2NF): 기본 키에 완전 함수적 종속이 되도록 테이블을 설계
- 제3정규형(3NF): 기본 키가 아닌 컬럼이 다른 기본 키가 아닌 컬럼에 종속되지 않도록 테이블을 설계
- 정규화는 데이터 중복을 줄이고, 데이터 무결성을 높이는 데 도움이 되지만, 지나치게 정규화된 설계는 조인 연산이 많아져 성능 저하를 초래할 수 있음
- 따라서, 실제 비즈니스 요구사항과 쿼리 패턴을 고려하여 적절한 수준의 정규화를 적용하는 것이 중요함

### 1NF (제1정규형)

- 컬럼 하나에는 값 하나

```python
class JobPost(SQLModel, table=True):
  id: int | None = Field(default=None, primary_key=True)
  title: str
  tags: str # "python,fastapi,sqlalchemy" 형태로 저장
```

- `tags` 컬럼에 여러 태그를 쉼표로 구분하여 저장하는 것은 정규화 원칙에 어긋남
- 이 경우, 태그마다 별도의 레코드로 분리하여 저장하는 것이 정규화된 설계임
- 예시:

```python
class JobPostTag(Base):
    __tablename__ = 'job_post_tags'
    job_post_id = Column(Integer, ForeignKey('job_posts.id'), primary_key=True)
    tag = Column(String, primary_key=True)
```

- 위 예시에서는 `job_post_id`와 `tag` 컬럼의 조합이 기본 키 역할을 하는 조인 테이블이 설계되어 있음
- 이 설계에서는 각 태그가 별도의 레코드로 저장되므로, 태그 검색, 추가, 삭제가 용이해지고, 데이터 무결성도 보장됨

### 2NF (제2정규형)

- 복합 키의 일부에만 의존하는 값이 있지 않나?

```python
class CompanyJobLink(Base):
    __tablename__ = 'company_job_links'
    company_id = Column(Integer, ForeignKey('companies.id'), primary_key=True)
    job_post_id = Column(Integer, ForeignKey('job_posts.id'), primary_key=True)
    company_name = Column(String, nullable=False)
    job_post_title = Column(String, nullable=False)
```

- 위 예시에서 기본 키는 `(company_id, job_post_id)`라는 복합 키다.
- 그런데 `company_name`은 사실 `company_id`만 알면 결정된다.
- `job_post_title`도 `job_post_id`만 알면 결정된다.
- 즉, 두 컬럼 모두 복합 키 전체가 아니라 키의 일부에만 의존하고 있으므로 2NF에 어긋난다.
- 이런 식으로 부모 테이블에 있어야 할 정보를 조인용 테이블에 같이 넣으면 중복과 불일치가 생기기 쉽다.
- 예시:

```python
class Company(Base):
    __tablename__ = 'companies'
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)

class JobPost(Base):
    __tablename__ = 'job_posts'
    id = Column(Integer, primary_key=True)
    title = Column(String, nullable=False)

class CompanyJobLink(Base):
    __tablename__ = 'company_job_links'
    company_id = Column(Integer, ForeignKey('companies.id'), primary_key=True)
    job_post_id = Column(Integer, ForeignKey('job_posts.id'), primary_key=True)
```

- 위 예시에서는 회사 이름은 `Company`, 채용공고 제목은 `JobPost`에 두고, `CompanyJobLink`에는 연결에 필요한 키만 남긴다.
- 이 설계에서는 부분 함수 종속이 제거되어 2NF를 만족하게 된다.

### 3NF (제3정규형)

- 이 값은 계산 결과 아닌가?

```python
class OrderItem(SQLModel, table=True):
  order_id: int
  product_id: int
  unit_price: float
  quantity: int
  total_price: float # unit_price * quantity
```

- `total_price`는 `unit_price`와 `quantity`로 계산 가능한 파생 값이다.
- 이런 계산 가능한 값을 그대로 저장하면 원본 값과 불일치가 생길 수 있다.
- 이 경우, `total_price` 컬럼을 제거하고, 필요할 때 계산하여 사용하는 것이 정규화된 설계임
- 예시:

```python
class OrderItem(Base):
    __tablename__ = 'order_items'
    order_id = Column(Integer, ForeignKey('orders.id'), primary_key=True)
    product_id = Column(Integer, ForeignKey('products.id'), primary_key=True)
    unit_price = Column(Integer, nullable=False)
    quantity = Column(Integer, nullable=False)
```

- 위 예시에서는 `total_price` 컬럼을 제거하고, 필요할 때 `unit_price * quantity`로 계산한다.
- 이 설계에서는 `total_price`가 데이터베이스에 저장되지 않고, 계산된 값으로 사용되므로, 데이터 무결성도 보장되고, 저장 공간도 절약됨

## 성능과 정규화의 관계

- 정규화가 먼저고 성능은 나중
- 성능은 인덱스, 캐싱, 쿼리 최적화 등으로 해결할 수 있지만, 데이터 무결성이 깨진 설계는 나중에 고치기 어려움
- 정규화된 설계는 데이터 무결성을 보장하고, 유지보수성을 높이는 데 도움이 되며, 성능 문제는 실제로 발생했을 때 구체적인 원인을 분석하여 해결하는 것이 좋음
