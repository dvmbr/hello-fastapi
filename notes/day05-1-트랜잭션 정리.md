# day05-1-트랜잭션 정리

- DB 세션은 `get_db_session`에서 생성/정리만 담당한다.
- 트랜잭션 경계는 쓰기 로직에서 `async with session.begin():`로 명시한다.

## 1. 트랜잭션 관리 방식 비교

### 기존 방식 (의존성에서 자동 commit/rollback)

```python
async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session() as session:
        try:
            yield session
        except Exception:
            await session.rollback()
            raise
        else:
            await session.commit()
```

장점:

- 라우터 코드가 짧다.
- 학습 초기에는 빠르게 구현 가능하다.

한계:

- 트랜잭션 경계가 코드에 잘 보이지 않는다.
- 읽기 요청에서도 의존성 종료 시점 로직에 commit 경로가 섞인다.
- 프로젝트가 커질수록 의도치 않은 commit 위험이 생긴다.

### 권장 방식 (세션 생명주기와 트랜잭션 분리)

```python
async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session() as session:
        yield session
```

쓰기 예시:

```python
async with session.begin():
    session.add(obj)
    await session.flush()
```

장점:

- 트랜잭션 경계가 명확하다.
- 읽기/쓰기 의도가 코드에서 분리된다.
- 리뷰, 테스트, 유지보수에 유리하다.

## 2. 트랜잭션이란 무엇인가

트랜잭션은 DB 작업을 하나의 단위로 묶는 개념이다.

핵심 동작:

- 모두 성공하면 `commit`
- 중간에 실패하면 `rollback`
- 즉, "전부 반영 또는 전부 취소"를 보장한다.

## 3. 왜 read에는 보통 명시 트랜잭션이 필요 없나

- read(`SELECT`)는 데이터를 변경하지 않는다.
- 일반 조회 API에서는 원자적으로 함께 반영할 쓰기 작업이 없다.
- 그래서 보통 `GET` 요청은 `session.begin()` 없이 조회만 수행한다.

## 4. read에도 트랜잭션을 쓰는 예외

- 여러 조회 결과를 같은 시점 스냅샷으로 강하게 맞춰야 할 때
- 잠금 기반 조회(`SELECT ... FOR UPDATE`)가 필요할 때
- 격리 수준 제어가 중요한 고급 시나리오일 때

실무 기본 규칙:

- `GET`은 보통 begin 없이 조회
- `POST/PATCH/DELETE`는 `session.begin()`으로 트랜잭션 명시

## 5. Async 스택 체크리스트

- 엔진: `create_async_engine`
- 세션: `AsyncSession`, `async_sessionmaker`
- 드라이버: SQLite는 `sqlite+aiosqlite`
- 라우터: `async def`
