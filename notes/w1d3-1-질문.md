## 오늘 물어본 내용 정리

### 1. 카운터 증가 로직

- 질문: router에서 auto_increment_todo_id() 호출 후 todo_id_counter를 바로 써도 되는가?
- 결론: 값 변수(todo_id_counter)를 import해서 직접 쓰면 복사 이슈가 생길 수 있으므로, 증가 함수의 반환값을 즉시 사용해야 안전하다.
- 권장 패턴:
  - todo_id = auto_increment_todo_id()
  - user_id = auto_increment_user_id()

### 2. response_model과 return 값 관계

- 질문: create_user에서 return new_user(dict) 해도 되는가?
- 결론: 가능하다. FastAPI가 response_model(UserResponse)로 검증/직렬화한다.
- 효과:
  - 스키마에 없는 필드(예: hashed_password)는 응답에서 제외된다.
  - 응답 형식이 UserResponse로 통일된다.

### 3. status_code 적용 방식

- 질문: status_code도 decorator에서 자동 적용되는가?
- 결론: 그렇다. Response 객체를 직접 만들어 반환하지 않는 한, decorator의 status_code가 적용된다.
- 예: POST create_user는 status_code=201로 응답된다.

### 4. DEBUG 모드와 Swagger

- 질문: 로컬에서 DEBUG 모드를 어떻게 켜는가?
- 실행 방법: DEBUG=true uv run uvicorn app.main:app --reload
- 참고: docs_url, redoc_url이 DEBUG 조건으로 설정되어 있어야 /docs, /redoc 노출이 제어된다.

### 5. 422가 발생하는 상황

- 결론: FastAPI/Pydantic 유효성 검증 실패 시 발생한다.
- 대표 케이스:
  - Query 제약 위반: skip < 0, limit 범위 위반
  - 타입 불일치: bool/int 필드에 잘못된 타입 입력
  - Body 제약 위반: 문자열 길이, 이메일 형식, enum 값 불일치 등

## 오늘 코드에서 중요한 변경 포인트

### 1. 라우터에서 전역 상태 직접 변경 제거

- todo/user 모두 라우터에서 전역 카운터 직접 증가 대신 storage 함수 사용으로 정리.
- 효과: import 복사 이슈 회피, 역할 분리 명확화.

### 2. Depends 기반 접근으로 일관성 확보

- todo 라우터에서 저장소 접근을 Depends(get_todo_db)로 통일.
- 효과: 테스트/확장 시 유리하고 구조가 깔끔해짐.

### 3. 목록 조회 검증 일관성

- user list에도 Query 제약(skip, limit)을 적용해 422 동작을 todo와 유사하게 맞춤.

## 핵심 한 줄 요약

- 라우터는 HTTP 처리에 집중하고, 상태 변경/데이터 관리는 storage 함수로 위임하는 구조가 가장 안전하고 유지보수에 유리하다.
