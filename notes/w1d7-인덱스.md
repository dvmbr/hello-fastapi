# 인덱스

인덱스는 데이터베이스에서 데이터를 **빠르게 찾기 위해 사용하는 자료구조**다. 흔히 책의 목차와 비슷하다고 비유한다.

## 인덱스의 동작 원리

- 대부분의 RDBMS는 **B-Tree(균형 이진 트리)** 구조로 인덱스를 구현한다.
- 인덱스가 없는 경우, 원하는 데이터를 찾으려면 테이블 전체를 처음부터 끝까지 순차적으로 검사(Full Table Scan)해야 한다.
- 인덱스가 있으면, 트리 구조를 따라 $O(\log n)$ 단계만에 원하는 데이터를 찾을 수 있다.

## 인덱스의 장점

- 대용량 데이터에서 SELECT/WHERE/ORDER BY 성능이 획기적으로 향상된다.
- 특정 컬럼(혹은 컬럼 조합)에 대해 빠른 탐색, 정렬, 중복 체크가 가능하다.

## 인덱스의 단점

- 인덱스가 많아질수록 INSERT/UPDATE/DELETE 성능이 저하된다(모든 인덱스를 갱신해야 하므로).
- 인덱스 자체도 디스크 공간을 차지한다.
- 잘못된 인덱스 설계는 오히려 성능을 떨어뜨릴 수 있다.

## 인덱스 설계 기준

- **실제로 자주 조회되는(WHERE/ORDER BY/JOIN에 자주 등장하는) 컬럼**에만 인덱스를 건다.
- AND로 묶여 자주 쓰이는 컬럼 조합은 복합 인덱스를 고려한다.
- INSERT/UPDATE/DELETE가 매우 빈번한 컬럼에는 인덱스를 남발하지 않는다.
- 너무 많은 인덱스는 오히려 전체 성능을 떨어뜨릴 수 있다.

## 인덱스의 종류

- **단일 인덱스(Single Index)**: 하나의 컬럼에만 인덱스를 생성
- **복합 인덱스(Composite/Multi-Column Index)**: 여러 컬럼을 묶어서 인덱스를 생성 (예: (a, b))
- **유니크 인덱스(Unique Index)**: 인덱스 대상 컬럼(조합)의 값이 중복될 수 없도록 보장
- **커버링 인덱스(Covering Index)**: 쿼리에서 필요한 모든 컬럼이 인덱스에 포함되어, 테이블을 추가로 읽지 않아도 되는 인덱스
- **풀텍스트 인덱스(Fulltext Index)**: 텍스트 검색(검색어 포함 여부 등)에 최적화된 인덱스
- **해시 인덱스(Hash Index)**: 해시 함수를 이용해 값을 빠르게 찾는 인덱스(일부 DB에서 지원)
- **비트맵 인덱스(Bitmap Index)**: 값의 종류가 적은 컬럼(예: 성별, 상태값 등)에 효과적

## 실행계획과 EXPLAIN/ANALYZE

- **실행계획(Execution Plan)**: DB가 쿼리를 실행할 때 어떤 방식(인덱스 사용, 테이블 전체 탐색 등)으로 처리할지 미리 분석한 계획
- **EXPLAIN**: 쿼리 앞에 붙여 실행계획을 확인하는 명령어 (예: `EXPLAIN SELECT ...`)
  - 어떤 인덱스를 사용하는지, 테이블을 몇 번 읽는지 등 확인 가능
- **ANALYZE**: 실제 쿼리를 실행해보고, 각 단계별로 얼마나 시간이 걸렸는지까지 보여줌 (PostgreSQL 등)
  - `EXPLAIN ANALYZE SELECT ...`
- 실행계획을 통해 인덱스가 실제로 사용되는지, 쿼리 성능의 병목이 어디인지 진단할 수 있음

---

인덱스를 생각없이 걸면 (인덱스 과다) 오히려 성능이 떨어질 수 있다. 인덱스는 데이터베이스가 데이터를 빠르게 찾을 수 있도록 도와주는 자료구조이지만, 너무 많은 인덱스가 있으면 데이터 삽입, 업데이트, 삭제 시마다 매번 모든 인덱스를 갱신해야 하므로 성능이 저하될 수 있다.
