## aiosqlite가 무엇인가

- SQLite를 async/await 형태로 사용하기 위한 비동기 드라이버다.
- URL에서 `sqlite+aiosqlite:///...` 형식으로 사용한다.
- FastAPI async 라우터와 함께 쓰면 이벤트 루프 블로킹을 줄이기 좋다.

## DB 파일 위치와 기존 storage 처리

- `database.py` 같은 설정 코드는 `app/` 안에 두는 것이 일반적이다.
- 실제 SQLite 파일(`database.db`)은 프로젝트 루트에 생성된다.
- 기존 `storage/`는 in-memory 저장소이므로 SQLModel DB로 전환 시 역할이 겹친다.
- 학습 단계에서는 점진 전환(기존 유지 + 새 라우터 실습)도 좋은 방법이다.

## 이 코드 2줄 의미

```python
engine: AsyncEngine = create_async_engine(DATABASE_URL, echo=True)
async_session = async_sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)
```

- 첫 줄: 비동기 DB 엔진 생성 (`echo=True`면 SQL 로그 출력)
- 둘째 줄: `AsyncSession`을 만들어내는 세션 팩토리 생성
- `expire_on_commit=False`: commit 후에도 객체 속성 접근이 끊기지 않게 함

## 지금 앱이 비동기 앱인가

- 예. 근거는 다음과 같다.
- 라우터 핸들러가 `async def`
- FastAPI + Uvicorn(ASGI) 기반
- DB도 `create_async_engine`, `AsyncSession`, `aiosqlite` 스택 사용

## get_db_session 트랜잭션 패턴

권장 패턴:

```python
async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session() as session:
        try:
            yield session
        except Exception:
            await session.rollback()
            raise
        else:
            await session.commit()
```

- 성공 시 commit
- 예외 시 rollback 후 재전파
- 이렇게 해야 에러를 숨기지 않고 트랜잭션도 안전하게 정리된다.

## raise는 무엇인가

- 예외를 발생시키거나, 잡은 예외를 다시 던지는 키워드다.
- `except` 내부의 단독 `raise`는 현재 예외를 그대로 상위로 전달한다.
- 무한 반복이 아니라 한 번 전파되는 흐름이다.

## @asynccontextmanager와 lifespan의 yield 의미

구조:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup
    ...
    yield
    # shutdown
    ...
```

- `yield` 이전: 앱 시작 시 실행(startup)
- `yield` 시점: 서버가 요청 처리 구간으로 진입(serving)
- `yield` 이후: 앱 종료 시 실행(shutdown)

정리:

- 데코레이터가 없으면 FastAPI가 기대하는 lifespan context manager 형태가 깨진다.
- `yield`가 없으면 startup/shutdown 경계를 만들 수 없어 정상 lifecycle 구성이 어렵다.

## TYPE_CHECKING이 무엇인가

- `TYPE_CHECKING`은 `typing` 모듈에서 가져오는 상수다.
- 실행(runtime)할 때는 `False`다.
- 타입 검사 도구(mypy, pyright)가 분석할 때는 `True`처럼 동작한다.

예시:

```python
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from app.models.job_post import JobPost
```

왜 이렇게 쓰는가:

- 타입 힌트 검사를 위해서는 import가 필요하다.
- 하지만 런타임에 서로 import하면 순환 import가 생길 수 있다.
- 그래서 타입 검사 시에만 import하고, 실제 실행 시에는 import를 생략해 안정성을 확보한다.

한 줄 요약:

- `TYPE_CHECKING`은 "타입 검사 때만 import 실행"하기 위한 안전 스위치다.

## from **future** import annotations 이 무엇인가

- 타입 힌트를 즉시 평가하지 않고, 나중에 해석하도록 미루는 설정이다.
- 서로 참조하는 모델(예: `Company` ↔ `JobPost`)에서 순환 참조 문제를 줄이는 데 도움이 된다.

왜 쓰는가:

- 아직 import되지 않은 타입을 힌트에 써도 런타임에서 바로 깨질 가능성이 줄어든다.
- `TYPE_CHECKING` 패턴과 함께 쓰면 런타임 안정성과 타입 검사 정확도를 같이 챙길 수 있다.

예시:

```python
from __future__ import annotations

class Company(SQLModel, table=True):
    job_posts: list["JobPost"]
```

한 줄 요약:

- `from __future__ import annotations`는 타입 힌트 해석 시점을 늦춰서, 모델 간 참조를 더 안전하게 만든다.
